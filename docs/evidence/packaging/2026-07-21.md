# Windows packager environment spike - 2026-07-21

## Scope

Performed a read-only/local-temporary readiness spike for the Phase 7 choice between direct Nuitka
and Qt's official `pyside6-deploy` wrapper. No packager dependency, compiler, SDK, certificate, or
system setting was installed or changed; no executable was built.

- Branch: `main`

## Environment findings

| Requirement/candidate | Result |
|---|---|
| `pyside6-deploy` launcher | Present in the project virtual environment |
| Nuitka Python module/launcher | Not installed |
| `vswhere.exe` | Not present at the standard Visual Studio Installer path |
| MSVC `cl.exe` | Not on `PATH` |
| Qt dependency reader `dumpbin.exe` | Not on `PATH` |
| GCC/MinGW compiler and `mingw32-make` | Not on `PATH` |

An isolated `%TEMP%` `pyside6-deploy --init --mode standalone` probe successfully generated a spec
after its temporary Python file was written as UTF-8 without BOM. The tool reported that `dumpbin`
was unavailable and produced a spec whose deployment package requirement was `Nuitka==4.0`.

The initial temporary probe used PowerShell 5.1 `Set-Content -Encoding UTF8`, which added a BOM and
exposed a `pyside6-deploy` AST-parser failure. The corrected no-BOM probe passed. Neither temporary
file is part of the application or committed delivery configuration.

## Official-tool context

Qt documents [`pyside6-deploy`](https://doc.qt.io/qtforpython-6/deployment/deployment-pyside6-deploy.html)
as its desktop deployment tool and a wrapper around Nuitka. The same documentation says the spec can
be version-controlled, distinguishes onefile from standalone mode, requires a Windows `.ico`, and
lists `dumpbin`/MSVC tooling for efficient Windows dependency discovery. Qt's
[Windows deployment guide](https://doc.qt.io/qt-6/windows-deployment.html) also requires testing on a
machine without Qt/compiler installations and calls out platform, style, image-format, and icon
plugins as relevant runtime dependencies.

## Decision

**No packager is selected yet.** Availability of the wrapper is not a measured build comparison.
The next authorized packaging experiment must:

1. create an isolated, disposable build environment rather than mutating the locked development
   environment;
2. pin the tested Nuitka version and record compiler/SDK identity;
3. start with `pyside6-deploy` standalone mode for inspectable files and startup-error diagnosis;
4. include both models, icon resources, licenses/notices, OpenCV/MediaPipe native dependencies, Qt
   platform/style/image/icon plugins, and the Visual C++ runtime where applicable;
5. record build time, artifact size/count, cold/warm launch, `--diagnose-install`, Safe Mode launch,
   camera/model failure recovery, and clean-machine install/uninstall behavior;
6. compare a direct Nuitka configuration only under the same pinned inputs and acceptance checks;
7. choose a delivery only after both results or a documented hard incompatibility.

Onefile mode is not the first candidate because extraction behavior makes missing native assets and
startup recovery harder to inspect. This is a test-order decision, not a final rejection.

## Current blocker

A measured standalone build is blocked on an explicitly provisioned isolated compiler/toolchain and
Nuitka environment. This is not a blocker for source checkout, the verified Python wheel, or Devpost
repository review, but it blocks any claim of a standalone Windows release.
