# MEYES Build Week demo runbook

Target duration: **2:35**. Hard stop: **under 3:00**.

This is an operator checklist for the human recording session. It does not assert that a video has
been recorded, reviewed, uploaded, or accepted by Devpost. The entrant must compare the final video
and submission against the current [Official Rules](https://openai.devpost.com/rules) and
[Hackathon Website](https://openai.devpost.com/) before submitting.

## 1. Prepare without activating output

1. Check out the exact intended `main` revision and run:

   ```powershell
   .\scripts\submission_preflight.ps1 -VerifyRemote -RunFullCheck
   ```

2. Record the displayed revision and current passing test count for the final Devpost fields.
3. Enable Do Not Disturb; close mail, chat, password managers, private terminals, and unrelated
   browser tabs. Remove bystanders, private files, licensed music, and unauthorized marks.
4. Use a clean desktop and a disposable text/scroll target. Keep no unsaved work in that target.
5. Confirm `Ctrl+Alt+Shift+F11` is available and physically rehearse it once.
6. Open MEYES and confirm the persistent state says `SAFE MODE` before camera startup.
7. Frame one face and the intended hands in even lighting. Do not start recording until camera and
   diagnostics are stable.

## 2. Record the bounded story

| Time | Picture | Required narration/evidence |
|---|---|---|
| 0:00-0:15 | MEYES opens in `SAFE MODE` | Ordinary-webcam, local, assistive productivity prototype; not a medical device. |
| 0:15-0:45 | Start camera, show Diagnostics, wink, temple tap/hold | Face/hand inference runs locally; frames are not intentionally stored or uploaded. |
| 0:45-1:15 | Show hands-free Smooth Pursuit live capture and held-out result | Follow the moving target, show live quality feedback, and state the result exactly. `Review Required` is not accepted and cannot enable pointer movement. |
| 1:15-1:50 | Disposable target plus Live Input checklist | Confirm the arm dialog, show one wink click and bounded temple scroll, then trigger the emergency shortcut and show `SAFE MODE`. |
| 1:50-2:15 | Test result, commit history, architecture/evidence | Show the current count/revision and the separate observation, intent, candidate, consent, and executor boundaries. |
| 2:15-2:35 | Application plus concise close | Explain human product/safety decisions and how Codex with GPT-5.6 accelerated implementation, review, tests, and QA. |

Do not cut away during the transition into or out of Live Input in a way that hides consent,
status, or the emergency return. Camera startup must visibly remain disconnected from OS output.

## 3. Calibration and live-output decision

- If the fit says `Review Required` or `Rejected`, show that honest result and omit gaze movement.
- Show gaze movement only if the exact current display has a saved `Accepted` envelope produced by
  a complete configured four-limit evidence policy. Do not loosen limits for the recording.
- Wink/temple output still requires the disposable target, explicit per-session arm confirmation, clear
  physical buttons/modifiers, registered emergency shortcut, and visible `LIVE INPUT` state.
- If any preflight, camera, calibration, or native-output step faults, return to Safe Mode and record
  the real recovery. Do not edit a failure into an apparent success.
- A safe fallback is to show Diagnostics plus its bounded fake primitive trace and explain that real
  output was deliberately omitted. Never claim the fake trace is operating-system input.

## 4. Claims allowed by current evidence

- Source checkout and isolated wheel verification on Windows; no standalone installer is claimed.
- Local MediaPipe input-media processing with the separately disclosed dependency-network boundary.
- Deterministic fake-boundary tests and guarded Windows `SendInput` implementation.
- Native 100% display evidence and managed Windows CI.
- Explicit, volatile modal consent; emergency and lifecycle release paths.

Do not claim universal accuracy, medical benefit, 125%/150% scale validation, enabled High Contrast
human validation, broad screen reach, representative live performance, signed delivery,
clean-machine installer acceptance, or completed Devpost submission.

## 5. Stop safely and review

1. Trigger `Ctrl+Alt+Shift+F11`; verify `SAFE MODE`; stop the camera; close MEYES normally.
2. Confirm the final edit is under 3:00 and contains understandable English audio, captions, or an
   English translation covering both the project and Codex/GPT-5.6 use.
3. Watch the entire export for private data, notifications, bystanders, copyrighted media, cuts that
   obscure safety state, and statements beyond the exact shown evidence.
4. Upload to YouTube with judge-accessible visibility, then test the final URL in an incognito or
   private browser session with no creator login.
5. Keep the video, repository, and testing path free and accessible through the required judging
   period. Record the final URL only after this check passes.

## 6. Final submission handoff

- Add the exact repository revision and current test result.
- Add the final YouTube URL and judge quickstart.
- Add the primary Codex task `/feedback` Session ID.
- Confirm entrant/team eligibility, rights, category, country, submitter type, and repository access.
- Recheck rules/updates/deadline immediately before submitting.
- Confirm Devpost displays **Submitted**, not Draft.
